---
name: Stale tsc incremental cache hides real errors
description: pnpm run typecheck can report 0 errors while real type errors exist, because of stale .tsbuildinfo incremental caches. Also documents a real single-root-cause cascading error case.
---

`pnpm run typecheck` in this monorepo uses `tsc --incremental` with `.tsbuildinfo` files per package (`artifacts/*/.tsbuildinfo`, `lib/*/tsconfig.tsbuildinfo`, `scripts/tsconfig.tsbuildinfo`). If a prior run cached a file as "already checked" and it wasn't touched in the current diff, tsc will skip re-checking it even if it actually has errors — so "0 errors" from `pnpm run typecheck` is not reliable proof the whole repo compiles.

**Why:** discovered when a typecheck reported 0 errors after several edits, but deleting all `.tsbuildinfo` files and re-running surfaced ~90 `SelectQueryError`/`Argument of type ... not assignable` errors across many Supabase-client hook/component files (`useCurrency.tsx`, `useNotifications.ts`, `useTenant.tsx`, `Navbar.tsx`, `SEO.tsx`, etc.). These looked like a widespread Database-generic inference problem, but the actual root cause was a single invalid option passed to `createClient()` in the shared `client.ts` (`realtime: { url: ... }` — not a valid `RealtimeClientOptions` field in `@supabase/supabase-js@2.109.0`/`@supabase/realtime-js@2.109.0`; there is no supported way to override the realtime endpoint via client options — it's derived internally from the client's base URL). That one bad option poisoned the inferred type of the exported `supabase` client, cascading into every consumer file. Fixing just that one file resolved all ~90 errors.

**How to apply:** (1) before trusting a "0 errors" typecheck result as proof a broad change didn't break anything, delete all `.tsbuildinfo` files in the repo and re-run `pnpm run typecheck` once for a true clean signal. (2) When a huge batch of type errors appears across many unrelated-looking files that all share one import (a shared client/context/hook), suspect a single root file first — fix its own type error, then re-check, rather than fixing each consumer file individually.
